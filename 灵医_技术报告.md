# 🎋 灵医 (LingYi) - 中医多智能体系统技术报告

## 1. 项目概述 (Project Overview)

“灵医” (LingYi) 是一款基于 **LangGraph** 框架驱动，以 **Qwen3** 系列大模型为核心推理引擎的垂直领域中医诊疗智能体。该系统深入模拟了真实老中医的诊疗流程，具备多轮“望闻问切”交互能力，并前沿性地融入了**按需 RAG (Retrieval-Augmented Generation) 知识检索**、**长短期记忆与上下文自适应压缩**，以及**双重配伍安全护栏**机制。

本项目实现了业务流程编排（Python）与医学专业逻辑（Markdown Prompt）的深度解耦，确保了中医智能体在复杂病案推理时的安全性、连贯性与有理有据。

## 2. 系统核心架构 (Core Architecture)

项目采用基于状态图（StateGraph）的多智能体协作架构，数据以 `AgentState` 的形式在各个专门能力的节点（Node）之间无缝流转。

### 2.1 状态流转模型 (State Management)
系统定义了严格且强类型化的 `AgentState`，涵盖了从前端交互到深层推理的完整生命周期数据：
- **基础对话**：`messages` 记录对话历史。
- **医疗上下文**：`symptoms` (提取的症状)、`input_files` & `parsed_files` (病历附件解析追踪)。
- **RAG 上下文**：`retrieved_docs` (文献片段)、`rag_score` (召回质量)、`rag_retry_count`。
- **临床推演**：`intent_type` (用户意图)、`diagnosis` (辨证)、`treatment_plan` (处方)。
- **安全与合规**：`safety_errors` (冲突报错)、`safety_violation_msg` (违规拦截消息)。
- **记忆与档案**：`patient_profile` (患者长效画像)、`summary` (记忆压缩摘要)。

### 2.2 核心工作流 (LangGraph Workflow)
整个系统的运行时受 `graph.py` 的 Master Router 协同控制，形成了一套完整的医疗级推演引擎：

1. **初始化与档案读取**：`Reader Node` 解析附加文件，`Mem Recall Node` 读取患者长期档案与既往处方。
2. **前置安全拦截**：`Safety Guard Node` 执行意图筛查，直接拦截违规配伍行为并强制终止。
3. **问诊与意图分流**：`Inquiry Node` 提取核心症状，辨别当前是“闲聊”、“追问”还是“请求开方”。
4. **按需 RAG 增强环路**：
   - 触发条件：当遇到非典型的复杂证型，需要古籍支持时。
   - 环路组件：`RAG Search` (古籍检索) -> `RAG Grader` (质量评估)。
   - 自我进化：若评估不达标，触发 `RAG Rewrite` 进行查询重写，最多重试 3 次。
5. **诊疗核心**：`Diagnosis Node` 输出理、法，`Treatment Node` 推荐方药与调摄建议。
6. **后置安全核验**：对生成的方药提取实体，与物理安全库进行十八反十九畏验证，若冲突则强制回炉重造。
7. **记忆落库与压缩**：`Summraizer Node` 判断上下文并在 Token 溢界前压缩，`Writer Node` 将有价值的病历档案持久化。

## 3. 关键技术亮点 (Technical Highlights)

### 3.1 双重医疗安全护栏 (Dual Safety Guardrails)
中医开方直接关系患者生命安全，项目通过两道防线彻底阻绝毒副方剂输出：
- **前置意图阻断 (Safety Guard)**：位于大模型正式推理前的环节。若用户由于缺乏常识恶意要求添加剧毒或相冲药材（如“帮我把甘草和海藻放一起开”），智能体立即触发拒绝响应机制，返回严厉警告并生成防违规记录。
- **后置配伍校验 (Treatment Safety Check)**：大模型生成的任何处方必须流经后置校验环节。通过实体抽取匹配 `tools/safety_rules.py` 里的严格代码逻辑，消除大模型的幻觉概率。一旦违规，向模型注入错误栈，要求模型即刻纠正重写。

### 3.2 动态按需知识检索 (On-Demand Self-Reflective RAG)
摒弃了传统的“逢问必查”的粗暴 RAG 模式：
- 引入了基于意图路由的精准投递，对于日常问候或典型基础病（感冒），直接利用模型强大的内化医学知识快速推理。
- 当触发检索时，依托 ChromaDB 对深度清洗过的经典 TCM 数据集（含《伤寒论》、《金匮要略》等）进行向量匹配。
- 引入 `RAG Grader`，使 Agent 具备自我反思能力：如果检索到的古籍条文无法解释当前患者病症，主动调用 `Rewrite` 模块转化白话到专业术语进行重检索。

### 3.3 永不遗忘的长期记忆与多线程状态 (Memory & Session Management)
- **Token 守门员**：配置了 `Summarizer`，一旦由于多轮问诊导致 Context 逼近上下文极限（如 8000 tokens），系统将自动执行无损压缩，将中间流水账合成“病历摘要”而不丢失病程主干。
- **患者专属画像库**：独立的 `Writer` 通过 SQLite 维护患者的“体质”、“过敏史”与“既往有效处方”。
- **无缝多线程会话 (Multithreading UI)**：Streamlit 结合 Cookie 缓存方案实现了工业级的鉴权流。借助 `Browser URL Query Params` 深度绑定 `thread_id`，允许用户随意刷新、随时多开历史“问诊记录”，记录切换即时复原历史推演拓扑状态。

## 4. 技术栈视图 (Tech Stack Detail)

| 组件类别 | 具体技术/框架 | 用途说明 |
| :--- | :--- | :--- |
| **工作流控制** | LangGraph, LangChain | 图形态流转、依赖注入、状态同步 |
| **基础模型层** | Qwen3 (API 或 vLLM 私有化部署) | 高层医学逻辑推理、意图识别与重写 |
| **向量知识库** | ChromaDB, BGE-embeddings | 文本存储、高维向量语义检索引擎 |
| **持久化存储** | SQLite3 (`profile_manager.py`) | 结构化档案、账户信息、对话线程管理 |
| **文档解析解析** | PyPDF2, python-docx | 结构化提取化验单与既往史文本 |
| **端侧界面层** | Streamlit, Extra-Streamlit-Components | 现代化问诊交互UI与无感知鉴权体系 |

## 5. 项目模块规范分析 (Code Organization)

系统完全依照面向对象与技能驱动范式（Skill-Driven）设计：
- `agent/skills/`：存放所有核心节点。严格执行 `.py`（业务流）+ `.md`（专家 Prompt）解耦。
- `agent/memory/`：管理状态压缩、会话重载点 `checkpointer`。
- `tools/`：确定边界规则（物理脚本），负责文件解析器和知识库调度。
- `app.py`：以高度整合的形式提供 UI 界面，负责 Token 透明流转与 Web 会话同步。

## 6. 总结 (Conclusion)
“灵医”采用最前沿的多智能体图计算（Graph-Based Multi-Agent）理论，攻克了常规中医 AI 中常见的“配伍幻觉”、“遗忘病史”、“知识不垂直”等痛点。不仅提供了可解释的推理链路，更通过多重内存架构和安全沙盒设计，成功构建了一个具备商用落地潜力的中医垂直领域专家级 Agent 基础底座。
