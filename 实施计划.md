# 🎋 灵医 (LingYi) 完整项目实施落地计划

本计划旨在指导开发者基于 **LangGraph** 和 **Qwen3** 构建具备长效记忆、多源文档解析及按需 RAG 能力的中医智能体。

## 0. 项目准备与技术栈

- **核心框架**: LangGraph (StateGraph), LangChain
- **大模型**: Qwen3-Max (API) / Qwen3-8B (本地 vLLM 部署)
- **数据库**: SQLite (画像与用户会话持久化), ChromaDB (向量检索)
- **前端界面**: Streamlit (集成 extra_streamlit_components 实现 Cookie 鉴权)
- **文件处理**: PyPDF2, python-docx
- **协议**: Model Context Protocol (MCP)

## 1. 第一阶段：基础设施与状态定义 (Day 1-2)

### 1.1 目录结构标准

确保项目按照以下结构初始化，严禁随意更改路径：

```
LingYi/
├── agent/
│   ├── memory/           # 记忆管理与摘要逻辑
│   ├── skills/           # 技能定义 (.py + .md)
│   ├── graph.py          # LangGraph 编排中心
│   ├── state.py          # 状态字典定义
│   └── prompts.py        # 全局静态 Prompt
├── TCM_data/             # 中医古籍清洗后数据
├── data/                 # 数据下载与预处理脚本
├── tools/                # 物理工具与规则 (含 document parser, RAG client)
├── storage/              # 持久化存储 (SQLite, ChromaDB)
├── model_provider.py     # 模型工厂
├── config.py             # 集中配置管理
└── app.py                # UI 界面与统一登录入口
```

### 1.2 状态机定义 (`agent/state.py`)

定义 `AgentState` 类，支持完整生命周期的数据流转：

```python
from typing import Annotated, List, TypedDict, Optional, Dict, Any
from langchain_core.messages import BaseMessage
from langgraph.graph.message import add_messages

class AgentState(TypedDict, total=False):
    messages: Annotated[List[BaseMessage], add_messages] # 自动合并消息
    symptoms: List[str]            # 结构化症状清单
    input_files: List[str]         # 上传的文件路径
    parsed_files: List[str]        # 跟踪已解析的文件，防重复
    extracted_file_content: str    # 文件纯文本内容
    retrieved_docs: List[str]      # RAG 检索到的文献片段
    intent_type: str               # 意图类型
    rag_retry_count: int           # RAG 重试计数
    rag_score: float               # RAG 评估得分
    diagnosis: Optional[str]       # 辨证结论
    treatment_plan: Optional[str]  # 诊疗方案
    safety_errors: Optional[str]   # 安全校验报错信息
    safety_retry_count: int        # 安全重试计数
    safety_violation_msg: Optional[str] # 违规消息记录
    patient_profile: Dict[str, Any] # 长期画像数据
    summary: str                   # 压缩后的背景摘要
    has_provided_treatment: bool   # 方案完备标记
```

## 2. 第二阶段：技能规格化与加载机制 (Day 3-4)

### 2.1 技能载入基类

在 `agent/skills/` 中实现一个通用逻辑，确保 `.py` 脚本能自动读取同名的 `.md` 说明文件作为 System Prompt。

### 2.2 问诊技能 (`Inquiry Skill`)

- **.md**: 定义问诊标准（必须询问寒热、汗、头身、舌脉等）。
- **.py**:
  1. 识别意图（判断用户是想看病还是闲聊）。
  2. 提取症状并存入 `state["symptoms"]`。
  3. 判定 `intent_type`。

### 2.3 文档解析技能 (`Doc Reader Skill`)

- **物理工具**: 开发 `tools/file_parser.py` 支持 PDF/Word/txt。
- **解析逻辑**: 仅提取与医学相关的指标（如化验单、既往处方），忽略行政信息。

## 3. 第三阶段：按需 RAG 优化环路 (Day 5-6)

### 3.1 检索决策逻辑

在 `graph.py` 中实现路由函数，判断是否需要启动 RAG。若症状为简单常见证（如单纯风寒感冒），跳过检索。

### 3.2 RAG 环路实现 (`rag_search.py`)

1. **QueryGen**: 将口语症状转化为中医术语检索词。
2. **Retrieve**: 通过 `vector_db_client.py` 检索古籍库。
3. **Grader**: 评估检索内容与当前病例的契合度（评分 0-1）。
4. **Rewrite**: 若评分低于 0.7 且重试 < 3 次，重写检索词。

## 4. 第四阶段：记忆管理与上下文压缩 (Day 7-8)

### 4.1 持久化存储与多线程流转

- 配置 `SqliteSaver` / 专有 DB 作为 LangGraph Checkpointer 和画像中心。
- 实现 User -> Thread 的多对多映射，允许用户刷新页面不丢失最近对话。
- 通过 URL 参数与 Browser Cookie 来自动同步 Streamlit 会话中的 `thread_id`。

### 4.2 上下文压缩 (`summarizer.py`)

- **触发条件**: 检测 `messages` 的 Token 总量。
- **阈值**: **8000 Token**（触发线设在 95% 即 7600 Token）。
- **动作**:
  1. 提取核心诊疗节点信息（症状、辨证结果）。
  2. 将冗长的历史对话压缩为一段“病历纪要”。
  3. 保留 System Message 和最近 3 轮对话，清空中间消息。

## 5. 第五阶段：安全审查与方案输出 (Day 9)

### 5.1 安全红线审查 (`Safety Guard`)

- **逻辑**: 方案生成后，提取药名。
- **物理校验**: 调用 `tools/safety_rules.py`（十八反十九畏）。
- **环路反馈**: 若不安全，将具体冲突药材反馈给 `Treatment` 节点，模型必须无条件删除该药材并重写。

### 5.2 长期画像更新 (`Writer Skill`)

当诊疗闭环结束时，将本次提取到的“体质”、“过敏史”增量写入 `profile.db`。

## 6. 第六阶段：总装与 UI 交付 (Day 10)

### 6.1 图编排 (`graph.py`)

完成所有节点注册与条件路由：

- `START -> Reader -> MemRecall -> SafetyGuard`
- `SafetyGuard -> (拒绝响应 -> SummarizeCondition | 验证通过 -> Inquiry)`
- `Inquiry -> CheckIntent -> (RAGSearch | Diagnosis | End)`
- `RAGSearch -> RAGGrader -> (Diagnosis | RAGRewrite -> RAGSearch)`
- `Diagnosis -> Treatment -> TreatmentSafetyCheck`
- `TreatmentSafetyCheck -> (重新Treatment | 处方安全 -> SummarizeCondition)`
- `SummarizeCondition -> (Token超限 -> Summarize | 否 -> Writer)`
- `Summarize -> Writer -> END`

### 6.2 界面实现 (`app.py`)

- 使用 Streamlit 实现带账户验证体系的多端界面。
- 支持 Cookie 免密登录及患者账户快速建档。
- 可随时新建或切换过往“对话记录”，支持会话重命名和删除功能（防空白记录污染）。
- 支持文件上传组件。
- 侧边栏实时展示 `Patient Profile` 状态。
